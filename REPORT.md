## 1. У чому полягають ключові відмінності між class і struct?
Класи (class) є reference type, а структури (struct) — value type.
Клас зберігається у heap і передається за посиланням, структура — у stack (коли можливо) і передається копіюванням.
Структури не підтримують успадкування, за замовчуванням не можуть бути null, працюють швидше для невеликих об’єктів і мають менше навантаження на GC.
## 2. Коли доцільно використовувати структури, а коли класи?
Структури варто використовувати, коли об’єкт невеликий, представляє значення (Point, DateRange), часто копіюється, має бути immutable, і важлива продуктивність.
Класи варто використовувати для складних моделей, об’єктів зі станом, поведінкою, життєвим циклом, або коли потрібне успадкування.
## 3. Що таке readonly struct і які переваги він дає?
readonly struct — це структура, всі поля якої є незмінними після створення.
Переваги: іммутабельність, безпечність у багатопотоковості, оптимізації компілятора, менше копіювань, вища продуктивність, менше помилок зі зміною стану.
## 4. Наведіть приклади реалізації Equals(), GetHashCode() та IEquatable<T> у ваших структурах.
Point:
public bool Equals(Point other) => X == other.X && Y == other.Y;

public override bool Equals(object? obj) =>
    obj is Point p && Equals(p);

public override int GetHashCode() =>
    HashCode.Combine(X, Y);

public static bool operator ==(Point a, Point b) => a.Equals(b);
public static bool operator !=(Point a, Point b) => !a.Equals(b);
GradeRecord:
public bool Equals(GradeRecord other) =>
    Subject == other.Subject && Grade == other.Grade && Date == other.Date;

public override int GetHashCode() =>
    HashCode.Combine(Subject, Grade, Date);
DateRange:
public bool Equals(DateRange other) =>
    Start == other.Start && End == other.End;

public override int GetHashCode() =>
    HashCode.Combine(Start, End);
## 5. Які результати ви отримали при порівнянні продуктивності struct vs class? Проаналізуйте їх.
У тестах продуктивності:

створення масиву структур було швидшим, ніж масиву класів;

сортування структур також було швидшим, бо вони компактніші;

структури споживали менше пам’яті, бо не створюють об’єкти в heap;

класи працювали повільніше через накладні витрати на посилання та GC.

Висновок: структури значно ефективніші для невеликих, часто копійованих об’єктів.
## 6. Як інтеграція структур вплинула на вашу систему?
система стала працювати швидше;

лабораторні місця (Point[]) стали легшими для обробки;

код став чистішим і логічнішим.
## 7. Які труднощі виникали під час роботи зі структурами?
правильна реалізація Equals() та GetHashCode();

помилки через копіювання значень замість роботи з посиланнями;

неможливість використання успадкування;

необхідність робити структури immutable;

boxing/unboxing у певних ситуаціях;

необхідність розуміння різниці між value та reference semantics.
